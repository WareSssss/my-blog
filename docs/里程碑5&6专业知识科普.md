# 里程碑 5 & 6 专业知识科普（新手友好版）

本文帮助你理解后续两个阶段要做什么，以及背后的关键概念与常见坑：
- 里程碑 5：AI 聊天（非流式 -> 流式可选）
- 里程碑 6：RAG（知识库检索增强生成）+ 引用来源展示

---

## 0. 先用一句话理解这两个里程碑
- 里程碑 5：让网站具备“能聊天”的 AI（把你的问题发给模型，拿到回答）
- 里程碑 6：让 AI “基于你的博客内容回答”，并且告诉你“引用了哪篇文章”（RAG + citations）

---

## 1. 里程碑 5：AI 聊天（Chat）要懂的核心知识

### 1.1 什么是“非流式”和“流式”？
- 非流式：你点发送 -> 等几秒 -> 一次性返回整段回答
  - 优点：实现简单，前后端都好做
  - 缺点：体验像“卡住”，长回答等待会更明显
- 流式：你点发送 -> 文字一边生成一边显示（像 ChatGPT）
  - 常见实现：SSE（Server-Sent Events）或 WebSocket
  - 优点：体验好、可中断、感知更快
  - 缺点：实现复杂一些（连接管理、断线重连、合并 token、错误处理）

建议落地顺序：
- 先非流式跑通完整链路（里程碑 5 先过）
- 再上流式（作为增强）

### 1.2 聊天系统里的“会话”和“消息”是什么？
一般会做两个核心概念：
- 会话（session）：一次聊天的容器（用来把消息串起来）
- 消息（message）：一条用户消息或 AI 回复

你现在数据库里已经有对应表（后续会正式用起来）：
- `chat_sessions`：记录 clientId、模型、标题等
- `chat_messages`：记录 role（user/assistant/system）、content、citations、token_usage 等

### 1.3 role：user / assistant / system 各是什么？
- system：给模型的“总规则/人格设定”（例如：你是扶桑的 AI 分身，回答要简洁）
- user：用户提问
- assistant：模型回复

关键点：
- system 信息通常要放在最前面（优先级更高）
- 永远不要把密钥放到前端（模型调用必须在后端做）

### 1.4 模型选择（model picker）是什么？
就是你在 UI 上看到的模型下拉框（例如 GLM、GPT 系列）。

工程上通常有两层：
- 前端：只展示可选项 + 把 modelId 传给后端
- 后端：真正决定用哪个 SDK/哪个 endpoint 调用

原因：
- 密钥在后端
- 你可以在后端做统一封装（不同厂商接口不一样）

### 1.5 限流（Rate Limit）为什么需要？
AI 接口贵，而且容易被刷。

最小限流粒度：
- 按 IP 限制请求频率
- 或按 clientId（前端生成一个匿名 ID）限制频率

你在文档里看到的建议（throttler）就是做这个。

### 1.6 超时、重试、降级文案
现实情况一定会遇到：
- 模型超时
- 模型限流
- SDK 报错
- 网络波动

必须准备：
- 超时（例如 30s）
- 重试（少量，避免无限重试）
- 降级文案（比如“当前服务繁忙，请稍后再试”）

### 1.7 输出结构（推荐统一格式）
后端返回给前端建议统一结构（无论是否 RAG）：
```json
{
  "answer": "……",
  "citations": [
    { "title": "文章标题", "url": "/blog/xxx", "snippet": "命中文段…", "score": 0.78 }
  ]
}
```
这样里程碑 5（纯聊天）和里程碑 6（RAG）不会打架。

## 2. 里程碑 6：RAG（知识库）要懂的核心知识

### 2.1 RAG 是什么？为什么需要？
RAG = Retrieval-Augmented Generation（检索增强生成）

一句话：
- 先从你的知识库里“检索”相关内容
- 再把检索结果当作上下文喂给模型生成回答

为什么需要：
- 解决“模型不知道你是谁/不知道你写了什么”的问题
- 减少胡编（hallucination）
- 可展示引用来源（让回答可追溯）

### 2.2 知识库构建的 3 件事
要让 RAG 能跑起来，至少要做这三件事：
- 切片（chunking）：把文章切成一段段（太长不行，太短也不行）
- 向量化（embedding）：把每个 chunk 转成向量
- 入库（vector store）：把向量和元数据存进向量库（Qdrant / Chroma 等）

你数据库里对应两张表：
- `kb_documents`：文档元信息（来自哪篇文章/链接是什么/是否启用）
- `kb_chunks`：切片内容（chunk_index、content、vector_store、vector_id、meta）

### 2.3 “向量”是什么？embedding 在做什么？
向量可以理解为“把文本变成一串数字”，用来做相似度检索：
- 语义相近的句子，向量会更接近

embedding 的本质：
- 输入文本 -> 输出向量（一个 float 数组）

### 2.4 TopK 召回是什么？
检索时通常会取最相关的前 K 条：
- K = 5/10/20 常见
- K 过大：上下文太长，成本高，反而干扰回答
- K 过小：容易漏掉关键内容

### 2.5 去重与拼接上下文（非常关键）
检索出来的 chunk 可能会：
- 重复（同一篇文章相邻段落都命中）
- 顺序混乱（缺少上下文）

常见做法：
- 同文档去重（保留得分最高的若干段）
- 按 chunk_index 排序
- 拼接成“Context”给模型

### 2.6 提示词模板（Prompt Template）
RAG 的提示词通常会明确要求：
- 只能基于提供的 Context 回答
- 如果 Context 不足，要明确说“不确定/没有命中”
- 输出 citations（让前端展示引用）

简单模板（示意）：
```text
你是扶桑的 AI 助手。
仅根据下面的 Context 回答问题。
若 Context 没有答案，请说明未命中知识库，不要猜测。

Context:
{context}

Question:
{question}

Answer（简洁）:
```

### 2.7 引用来源（citations）怎么生成？
每个 chunk 入库时就要带上元数据：
- title（文档标题）
- url（文章链接）
- snippet（命中的文本片段）
- score（相似度得分）

返回给前端后，UI 就能展示“引用来源”卡片。

### 2.8 RAG 常见坑（新手必看）
- 切片太长：上下文爆炸、成本高、回答变差
- 切片太短：缺少语境，回答不准确
- 不做去重：重复内容喂给模型，回答更混乱
- 不做兜底：未命中时胡编，用户信任崩掉
- 直接把整篇文章塞上下文：最容易超上下文窗口

## 3. 里程碑 5 & 6 的推荐实现顺序（最稳）
1) 先完成里程碑 5 的“非流式聊天”
   - 前端：消息列表 + 输入框 + 模型选择
   - 后端：/chat（或 /public/chat）接口调用云模型
   - 存储：写入 chat_sessions/chat_messages
2) 加入最小限流与错误处理
3) 再上 SSE 流式（可选增强）
4) 再进入里程碑 6：RAG
   - 构建索引（从 posts 导入 kb_documents/kb_chunks + 向量库）
   - chat 接口增加“先检索再生成”
   - 返回 citations 并在前端展示

## 5. 新手建议：你需要先掌握哪些最小概念？
- HTTP 请求：前端怎么调后端接口
- JSON：请求体/响应体结构
- 环境变量：密钥放哪里（永远不进前端）
- 数据库表：session/message/document/chunk 的意义
- “先跑通再优化”：非流式 -> 流式；纯聊天 -> RAG
