# AI Token 消耗治理与安全防护方案

本文档详细描述了在博客项目上线前，针对 AI 接口进行的 Token 消耗控制、成本管理以及安全防护策略。

---

## 1. 核心治理策略

### 1.1 请求频率治理 (Rate Limiting)
- **IP 级别限制**：限制单个 IP 每分钟/每小时的请求次数（例如：每分钟 5 次，每天 50 次）。
- **客户端 ID 绑定**：结合 `clientId` 进行多维度限流，防止通过代理 IP 绕过限制。
- **策略分级**：对“新建会话”设置更高的限制权重，对“流式输出”设置连接数限制。

### 1.2 上下文窗口管理 (Context Management)
- **固定窗口滑动**：目前项目已实现 `take: 10`（仅取最近 10 条消息）。建议根据模型特性缩减至 6-8 条。
- **Token 计算与截断**：在发送前预估 Token。如果上下文超过阈值，则强制丢弃最早的对话。
- **系统提示词优化**：精简 System Prompt，去除冗余描述。

### 1.3 响应长度限制 (Response Control)
- **Max Tokens 约束**：显式设置 `max_tokens` 参数（建议设置为 1000-1500），防止模型生成过长回答。
- **停止符设置**：合理使用 `stop` 标记，让模型在完成回答后立即结束。

---

## 2. RAG 检索优化 (RAG Cost Control)

- **Top-K 限制**：严格控制检索出的片段数量（目前建议 Top-3）。
- **片段长度控制**：将 Markdown 切片控制在 500-800 字符以内。
- **语义过滤**：如果检索出的内容相似度分值低于阈值（如 < 0.7），则不发送给 AI。

---

## 3. 架构层防护措施

### 3.1 语义缓存 (Semantic Cache)
- **原理**：对于重复或高度相似的问题，直接从缓存返回，不调用 AI。
- **实施**：使用向量匹配常见问题的 Embedding，命中则直接返回结果。

### 3.2 验证码机制 (Proof of Work)
- **人机验证**：连续对话超过阈值后弹出验证码。
- **PoW 挑战**：要求浏览器完成一段简单的计算，增加恶意攻击成本。

---

## 4. 成本监控与预警

- **Token 审计**：在数据库记录每条消息消耗的 `prompt_tokens` 和 `completion_tokens`。
- **预算熔断**：
    - **每日阈值**：设置每日总消耗上限。
    - **余额监控**：定期检查 API 余额，低于预警线时发送告警。

---

## 5. 落地建议 (Action Items)

1. **配置参数**：在 `.env` 中增加 `AI_MAX_TOKENS_PER_DAY` 和 `AI_MAX_CONTEXT_MESSAGES`。
2. **完善限流**：在后端 `app.module.ts` 中配置精细化的 `ThrottlerModule`。
3. **数据分析**：上线后定期审计 `chat_messages` 表中的 `tokenUsage` 字段。
