# 个人博客产品需求文档（PRD）

## 1. 文档信息
- 产品名称：个人博客
- 文档版本：v1.0
- 创建日期：2026-04-18
- 文档目标：明确个人博客产品的业务目标、功能范围、交互逻辑和交付标准，指导后续设计与开发。

## 2. 产品背景与目标
### 2.1 背景
个人需要一个统一的线上主页与内容阵地，用于展示个人能力、持续输出技术内容，并提供实用开发工具与 AI 问答能力，提升个人品牌影响力和访问者留存。

### 2.2 产品目标
- 对外展示个人资料、技术栈、项目经历与联系方式。
- 沉淀并分类输出博客内容（技术文章、学习笔记、项目分享）。
- 提供常用在线开发工具集合，提高访问者开发效率。
- 集成基于 RAG 知识库的 AI 聊天能力，支持智能问答。

### 2.3 成功指标（MVP）
- 首页信息完整度：核心模块展示率 100%。
- 内容体系可用：博客分类可浏览、可阅读、可检索。
- 工具页可用：至少上线 8-12 个常用工具入口。
- AI 问答可用：支持基础多轮问答，回答有来源上下文。

## 3. 用户与使用场景
### 3.1 目标用户
- 招聘方/合作方：快速了解你是谁、会什么、做过什么。
- 技术同行：阅读你的技术文章与项目实践。
- 你本人：沉淀知识、复盘学习、展示成长轨迹。

### 3.2 核心场景
- 新访客首次进入，快速了解个人信息并建立信任。
- 读者按分类查阅技术文章和学习笔记。
- 开发者进入工具页快速找到常用在线工具。
- 用户在 AI 聊天区提问，获取基于知识库的回答。

## 4. 产品范围
### 4.1 本期（MVP）范围
- 首页（个人介绍页）
- 博客文章模块
- 开发工具集合模块
- AI 聊天模块（RAG）
- 基础管理能力（文章发布、分类管理、工具配置）

### 4.2 暂不纳入（后续迭代）
- 用户注册登录与评论系统
- 多语言国际化
- 复杂推荐系统
- AI 历史会话云同步

## 5. 信息架构（IA）
- 首页
- 博客
- 开发工具
- AI 聊天
- 关于/联系方式（可并入首页）

建议顶部导航固定：`首页 | 博客 | 开发工具 | AI聊天`

## 6. 功能需求明细
### 6.1 首页（个人资料与能力展示）
#### 6.1.1 模块结构
- Hero 区：姓名/身份一句话介绍 + 头像 + CTA（查看博客、联系我）。
- 个人资料：教育/工作年限/当前方向/兴趣方向。
- 技术栈：按前端、后端、数据库、工程化、AI 等维度展示。
- 主要经历：关键项目或成果卡片（项目名、角色、结果、技术点）。
- 联系方式：邮箱、GitHub、掘金/知乎/公众号等。

#### 6.1.2 功能要求
- 支持移动端与桌面端自适应。
- 技术栈和项目经历支持后台配置。
- 联系方式支持一键复制（邮箱）与外链跳转。

#### 6.1.3 验收标准
- 首屏 3 秒内可识别“你是谁、做什么”。
- 访问者 1 次滚动内可看到技术栈与核心经历。
- 联系方式至少包含 2 种可用渠道。

### 6.2 博客文章模块
#### 6.2.1 分类要求
- 技术文章
- 学习笔记
- 项目分享

#### 6.2.2 列表页功能
- 按分类筛选。
- 按发布时间倒序展示。
- 文章摘要、标签、阅读时长展示。
- 关键词搜索（标题/摘要）。

#### 6.2.3 详情页功能
- 标题、发布时间、分类、标签、正文。
- 支持 Markdown 渲染（代码高亮、图片、引用）。
- 目录导航（TOC）和返回顶部。

#### 6.2.4 管理侧需求（最小可用）
- 创建/编辑/删除文章。
- 分类管理（固定三大类，可扩展）。
- 草稿与发布状态。

#### 6.2.5 验收标准
- 三大分类均可独立访问与筛选。
- Markdown 文章渲染正确，无排版错乱。
- 搜索结果与关键词匹配准确。

### 6.3 开发工具集合模块
#### 6.3.1 目标
聚合常用在线开发工具，减少开发者找工具成本。

#### 6.3.2 工具展示结构
- 分类：格式化、编码转换、网络调试、效率辅助、文档查询。
- 工具卡片：名称、简介、标签、外链地址、收藏/常用标记（可选）。

#### 6.3.3 建议首批工具清单（示例）
- JSON 格式化/校验
- Base64 编解码
- URL 编解码
- 时间戳转换
- 正则表达式测试
- JWT 解析
- HTTP 状态码速查
- Cron 表达式生成器
- SQL 格式化
- Markdown 在线预览

#### 6.3.4 管理侧需求
- 工具增删改查
- 分类与排序管理
- 链接有效性定期检查（可后续实现）

#### 6.3.5 验收标准
- 工具页可按分类筛选并快速跳转。
- 每个工具卡片信息完整且链接可用。

### 6.4 AI 聊天模块（RAG）
#### 6.4.1 产品定位
提供基于你个人知识库的智能问答，突出“懂你的内容”的回答质量。

#### 6.4.2 用户侧功能
- 输入问题并获取回答。
- 支持多轮对话（同会话上下文）。
- 回答中展示引用来源（文章标题/片段）。
- 异常提示（如知识库暂无相关内容）。

#### 6.4.3 后台与系统能力
- 知识库文档导入（Markdown/PDF/TXT，MVP 可先 Markdown）。
- 文档切片、向量化、索引构建。
- 检索 + 大模型生成（RAG）流程。
- 提示词模板可配置（系统角色、回答风格）。

#### 6.4.4 非功能要求（MVP）
- 首次响应建议 <= 5 秒（视模型与部署情况）。
- 对于无依据问题，优先回答“基于当前知识库暂无信息”。
- 不返回敏感配置信息（密钥、后台地址等）。

#### 6.4.5 验收标准
- 能基于知识库内容正确回答常见问题。
- 至少返回 1 条可追溯引用来源（有命中文档时）。
- 空结果场景有友好提示，不出现系统报错文案。

## 7. 关键页面与流程
### 7.1 首页流程
- 用户进入首页 -> 浏览个人信息与能力 -> 点击博客/联系方式/AI 聊天。

### 7.2 博客阅读流程
- 进入博客列表 -> 分类筛选/搜索 -> 打开详情页 -> 深度阅读。

### 7.3 工具使用流程
- 进入工具页 -> 按分类查找 -> 点击工具卡片跳转使用。

### 7.4 AI 问答流程（RAG）
- 用户提问 -> 系统向量检索相关文档 -> 生成答案并附来源 -> 用户追问。

## 8. 非功能需求
### 8.1 性能
- 首页首屏加载时间目标 <= 2.5 秒（常规网络）。
- 图片资源压缩与懒加载。

### 8.2 可用性
- 全站响应式适配（手机、平板、桌面）。
- 关键操作有反馈（加载中、成功、失败）。

### 8.3 SEO（博客重点）
- 每篇文章独立 URL、标题与描述可配置。
- 结构化标题层级清晰（H1-H3）。
- 生成 sitemap（后续可加 RSS）。
- 博客详情页采用预渲染（构建期生成静态 HTML）以提升收录与访问速度。

### 8.4 安全
- 后台接口鉴权采用 JWT + Refresh Token，不使用固定口令直连后台。
- API 密钥使用环境变量管理，不写入前端代码。
- 管理端登录接口增加限流与基础审计日志。

## 9. 技术方案建议（React + Nest 单路线）
### 9.1 总体架构
- 架构模式：前后端分离（Public Web + Admin API + AI/RAG Service）。
- 前端：React + TypeScript（负责首页、博客、工具页、AI 聊天页）。
- 后端：NestJS（负责内容管理、工具管理、鉴权、AI 问答编排）。
- 数据层：PostgreSQL（业务数据）+ 向量库（RAG 检索）。
- 部署形态：前端静态托管 + 后端 API 服务独立部署。

### 9.2 前端实现（React）
- 技术栈：React + Vite + TypeScript + React Router + Tailwind CSS。
- 内容渲染：Markdown 渲染（`gray-matter + remark/rehype + shiki/prism`）。
- SEO 策略：
- 博客详情页采用预渲染（构建期生成静态页面）。
- 页面补充 `title/description/og` 元信息与结构化标题。
- 自动生成 `sitemap.xml` 与 `robots.txt`。
- 数据交互：通过 REST API 访问 Nest 服务（Axios/Fetch 均可）。

### 9.3 后端实现（NestJS）
- 模块划分：`auth`、`content`、`tools`、`chat`、`admin`。
- 鉴权机制：管理员登录 + JWT Access Token + Refresh Token。
- 安全策略：登录限流、敏感接口鉴权、环境变量管理密钥。
- API 约定：统一响应格式、错误码、分页参数规范。

### 9.4 内容管理策略
- MVP 阶段采用 Markdown 文件驱动作为文章主数据源（`posts/` 目录）。
- Nest 负责文章索引、标签聚合、搜索接口与发布状态管理。
- 后续如需可升级为 CMS（Strapi），保持 API 层兼容。

### 9.5 AI 与 RAG（云模型）
- 模型策略：使用云端大模型 API（如 OpenAI/Claude/通义）。
- 检索流程：问题向量化 -> 向量库召回 -> 拼接上下文 -> LLM 生成 -> 返回答案+引用。
- 向量库建议：
- MVP：Chroma 或 Qdrant（部署简单）。
- 扩展：Pinecone/Weaviate（云端托管更方便）。
- 稳定性策略：超时控制、失败重试、降级文案、基础限流。
- 结果要求：命中时返回引用来源；未命中时明确提示“知识库暂无足够信息”。

### 9.6 数据库与核心模型
- 数据库：PostgreSQL（MVP 与长期统一，避免迁移成本）。
- 核心表建议：`posts`、`categories`、`tools`、`chat_sessions`、`chat_messages`、`kb_documents`、`kb_chunks`。
- 缓存扩展：二期引入 Redis（缓存热点查询与会话上下文）。

### 9.7 部署与工程化
- 前端部署：Vercel / Netlify（静态资源托管）。
- 后端部署：Railway / Render / 云主机 Docker。
- CI/CD：GitHub Actions（Lint、Build、Test、Deploy）。
- 代码规范：ESLint + Prettier + Husky。
- 环境管理：`.env.development`、`.env.production` 分离，密钥仅后端持有。

### 9.8 技术选型结论
- 本项目定版为：`React + Vite + TypeScript + NestJS + PostgreSQL + 云模型API + 向量库(RAG)`。
- 不采用 Nuxt/Next 双路线，减少学习与维护成本。
- 通过“博客预渲染 + SEO 基础配置”满足对外展示站点的收录需求。

## 10. 里程碑计划（建议）
- 第 1 周：需求确认、页面原型、技术方案定稿。
- 第 2 周：首页 + 博客列表/详情 + Markdown 渲染。
- 第 3 周：工具集合页 + 后台基础配置能力。
- 第 4 周：RAG 聊天接入 + 联调测试 + 上线发布。

## 11. 风险与应对
- RAG 回答不稳定：通过提示词优化、召回阈值调整、知识库清洗应对。
- 内容更新成本高：优先采用 Markdown 工作流与模板化。
- 工具链接失效：增加定期巡检机制（后续自动化）。

## 12. MVP 验收清单
- 首页：个人资料、技术栈、主要经历、联系方式完整上线。
- 博客：三大分类、列表筛选、详情阅读、搜索可用。
- 开发工具：首批工具集合可访问、分类清晰、跳转可用。
- AI 聊天：可提问、可回复、可展示引用来源、异常可提示。
- 全站：移动端适配与基础 SEO 满足上线标准。
